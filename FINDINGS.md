# NIKI — Findings: the product describes itself

**Phase 4 of the build plan** answers the open questions from the 10× research
with *data NIKI itself produces*, not marketing claims. Every number below is
generated by shipped code and is reproducible from this repo. Where the honest
answer is "not yet provable," we say so.

> **Positioning discipline.** We do **not** claim "10×." The research found no
> benchmark supports a literal 10×. What NIKI *can* prove is that it is
> **safer, more auditable, more economically transparent, and genuinely
> independent** than a single-agent tool — and each of those is demonstrated
> below, in NIKI's own output.

---

## P3.1 — Does the Reviewer genuinely challenge, or just ratify?

**Answer: it genuinely challenges — and we can measure the difference.**

NIKI injects an adversarial **Red** agent immediately before the Reviewer
(Slice 1.2). The Red agent's only job is to attack the Coder's diff; the
Reviewer must then *reconcile every Red challenge* (uphold → request revision,
or refute → justify). This structurally prevents the sycophantic "looks good to
me" convergence a single agent falls into.

**The data (Slice 1.3 eval harness, `niki eval`, deterministic replay):**

| Metric | NIKI (Red/Blue) | Baseline (single reviewer) |
|---|---|---|
| Catch rate (seeded defects) | **100%** | 50% |
| Reviewer false-approvals | **0** | 1 |
| False-approval reduction | — | **100%** |

The decisive case is `defect-sql`: a SQL-injection the **single-reviewer
baseline rubber-stamped** (`verdict: approved`, 0 issues) but NIKI caught via an
**upheld Red challenge** (`caught_by_red: true`, `red_upheld: 1`,
`verdict: revision_needed`). The `defect-offbyone` case is caught by both — so
the harness also demonstrates NIKI isn't just flagging everything.

**Reproduce:** `cargo run -- eval` → writes `.niki/eval/eval_report.{md,json}`.
Runs offline against recorded artifacts (`evals/fixtures/`), so the result is
CI-stable and needs no API keys. Use `--live` to drive real models.

**Honesty note:** this is a *seeded-defect smoke dataset* (2 cases), not
SWE-bench at scale. It proves the *mechanism* works and is measurable; it does
not yet claim a population-level catch-rate. Scaling the dataset is the next
evidence step.

## P3.2 — What is the real token multiple vs a single agent?

**Answer: transparently reported per run, and now adaptively minimized.**

Multi-agent independence has a cost: each agent re-ingests shared context
(spec + diff + tests) in its own session. NIKI never hides this. Every
`report.md` includes a **`### vs a single autonomous agent`** block (Slice 2.3)
that reports:

- **NIKI total tokens** vs the **single-agent estimate** (peak input + total
  output — what one agent would spend ingesting context once).
- The **token multiple** (`N×` — "the cost of isolation"), shown even when no
  pricing is configured, so the tax is always visible.
- The **redundant context re-ingested** across stages, in tokens.
- When cost is measured: NIKI `$`, the single-agent `$` estimate, and the
  **extra paid for multi-agent independence**.
- The **cheap-model-for-Tester control**: if the Tester runs a cheaper model
  than the Coder, NIKI names the saving; if all stages share a model, it flags
  the untapped saving.

**And NIKI now stops paying the tax when it isn't worth it (Slice 3.2).**
Adaptive topology auto-selects a **single-agent fast-path** for
bounded/sequential low-complexity tasks — collapsing Coder/Tester/Reviewer/Red
into one session, which drives the token multiple toward **1×** for exactly the
tasks where independence adds no value. The multi-agent chain is reserved for
work that genuinely benefits (or when security/parallel modes are on). The
selected topology is printed in every report's `## Pipeline Result` block.

**Net answer:** the token multiple is *not a fixed penalty* — it is reported
honestly per run and adaptively reduced to ~1× where the task doesn't need
independence.

## P3.3 — Does the moat hold vs Codex/Cursor-style isolated multi-agent?

**Answer: yes — the moat is safety + audit + economics, not "we're multi-agent."**

"We use multiple agents" is not a moat; competitors can add agents. NIKI's
defensible differentiators are structural and *provable in its own output*:

1. **Hermetic safety (Slice 1.1) — the strongest, most provable claim.**
   NIKI never mutates your working tree. Before/after a run it fingerprints the
   repo and emits a `SafetyProof` (`artifacts/safety_proof.json` + a
   `## Hermetic Safety Proof` report section) asserting: every pre-existing
   branch tip unchanged, exactly one new `niki/<id>` branch added, that branch
   parented on the base commit (no history rewrite), working tree byte-identical.
   This directly answers the research's strongest finding — competitor agents
   running in the live environment have caused **catastrophic data loss** (a
   Claude-powered agent deleted an entire company database in seconds). NIKI
   *shows* your tree was never touched.

2. **Audit trail that catches (Slice 2.2).** The report doesn't label itself
   "auditable" — it *proves* it, rendering each defect the Reviewer/Security
   Auditor caught as an **annotated diff** (`▶` on the exact offending line),
   tallied by severity. If nothing was caught, the section is empty — the claim
   is never fabricated.

3. **Economic transparency (Slice 2.3 + 3.2).** See P3.2 — NIKI is the tool
   that *tells you* its own token tax and then minimizes it, rather than hiding
   the multi-agent cost.

4. **Genuine context isolation (Slice 2.1).** The report states, per agent,
   exactly which published artifacts it saw and that it saw **no other agent's
   reasoning** — with a one-paragraph "why this isn't just subagents"
   (context-sharing subagents inherit the parent conversation and converge;
   NIKI's agents exchange only typed artifacts and cannot collude).

The moat is the **combination**: a competitor would have to replicate hermetic
proof *and* catch-annotated auditing *and* cost transparency *and* true context
isolation — each of which NIKI demonstrates in the artifact it hands you.

## P3.4 — Does the segment run local Docker, or expect managed cloud?

**Answer: by design, both — split along the revenue model. The empirical
segment mix is pending beta data; the intended model is now decided.**

The product model resolves the "local vs cloud" question deliberately rather
than leaving it to chance:

- **Free / self-hosted tier → local, BYOK.** We expect the **majority** of users
  to run their own infrastructure (Docker or `git worktree`, bring-your-own-key).
  NIKI runs fully locally today and does **not** require cloud to deliver its
  core value.
- **Enterprise tier → managed cloud.** A separate paid model where NIKI provides
  all server infrastructure, letting enterprise users run an effectively
  **unlimited** number of agents and use cases without owning the compute. Cloud
  is a **paid/enterprise surface**, not a free-tier feature.

The cloud backend is therefore the **enterprise revenue model**, not a
convenience layer. The `Sandbox`-trait seam (`src/sandbox/cloud.rs`) is ready for
that design to drop in, with the hermetic `SafetyProof` computed client-side so
trust never depends on the server.

**What remains pending:** the *empirical* mix — how many enterprise accounts opt
for managed cloud vs run BYOK internally — needs real beta usage data. That is
blocked on the deferred private beta (3.4) and cloud build-out (3.1), both slated
for the GTM/launch push. The **model** is decided; the **measured segment split**
will be published once the beta is live.
